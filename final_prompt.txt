# Build prompt: Event-Driven Congestion Forecasting & Response Simulator

Paste everything below this line into your coding assistant (e.g. Claude Code) along with the dataset file. It is written as a complete build brief — phases, formulas, file structure, and acceptance criteria — so it can be executed end to end without further clarification.

---

## 1. Context

You are building a hackathon prototype for the problem statement **"Event-Driven Congestion (Planned & Unplanned)."** Political rallies, festivals, construction, and sudden incidents (breakdowns, accidents, tree falls, water logging) create localized traffic breakdowns in Bengaluru. Today, event impact is not quantified in advance, resource deployment is experience-driven, and there is no post-event learning loop.

You will be given a CSV of historical traffic events (~8,000 rows) from Bengaluru Traffic Police with columns including: `id, event_type (planned/unplanned), latitude, longitude, endlatitude, endlongitude, address, event_cause, requires_road_closure, start_datetime, end_datetime, status, priority, corridor, zone, police_station, junction, veh_type, veh_no, description, closed_datetime, resolved_datetime`, plus several mostly-empty workflow columns (`client_id`, `*_by_id`, `citizen_accident_id`, `meta_data`, etc.).

**Build a system that does three things:**
1. Predicts how long an incoming event will disrupt traffic and how much road capacity it removes.
2. Simulates how that capacity loss propagates into a congestion queue using a real traffic-flow model (not a guess) — this is the centerpiece, it must be visibly a simulation, not just a number.
3. Converts the simulation output into a concrete resource plan: number of officers, number of barricades, and a recommended diversion route.

Wrap all three in an interactive dashboard that can replay historical days and let a judge click an incident to watch the full pipeline run live.

---

## 2. Phase 0 — Setup

- Load the CSV, print `.info()` and `.describe()`, confirm row count and column dtypes.
- Parse all `*_datetime` columns to UTC-aware timestamps.
- Create `event_end_time = COALESCE(end_datetime, closed_datetime, resolved_datetime)`. This is your real "when did disruption stop" field — `end_datetime` itself is populated for under 3% of rows, so don't rely on it alone.
- Create `duration_minutes = (event_end_time - start_datetime) / 60`, only where `event_end_time` is not null. Flag `status == 'active'` rows separately — their true duration is unknown (right-censored), do not treat them as a normal numeric label.
- Drop or ignore columns that are pure workflow metadata and carry no traffic signal: `client_id`, `created_by_id`, `last_modified_by_id`, `assigned_to_police_id`, `citizen_accident_id`, `map_file`, `meta_data`, `kgid`, `gba_identifier`.
- Treat `endlatitude`/`endlongitude` values of exactly `0` as missing, not as real coordinates.

## 3. Phase 1 — Data cleaning

- Standardize `event_cause`, `priority`, `status`, `corridor`, `zone` to consistent casing/categories; print value counts for each so inconsistent labels are visible before encoding.
- Impute missing `corridor` using nearest-neighbor match on `(latitude, longitude)` against rows that do have a corridor value (small fixed vocabulary, e.g. `ORR East 1`, `Bellary Road 2`, `Bannerghata Road`, `Hosur Road`, `Magadi Road`, `Tumkur Road`, `Mysore Road`, `Non-corridor`). This is your usable road-network labeling — treat it as a categorical feature, not free text.
- For `description`, build a small keyword tagger (regex/dictionary, English + transliterated Kannada terms you can see in the data, e.g. "puncture," "tyre," "starting problem," "water logging," "tree fall") to sub-classify the large `others` bucket where possible. Do not invest in a translation model — keyword matching is sufficient for this dataset's scale.
- Log a data-quality summary (rows kept/dropped, % missing per retained column, count of `active` vs `resolved`/`closed`) into `reports/data_quality.md` so it can be shown to judges as evidence of due diligence.

## 4. Phase 2 — Feature engineering

Build a feature table keyed by `id` with at least:
- Temporal: hour of day, day of week, is_weekend, is_peak_hour (7–10am, 5–8pm IST).
- Spatial: `corridor`, `zone`, `junction` (categorical), plus a binned location cluster (KMeans on lat/long, k≈15–25) as a fallback when corridor is missing.
- Event: `event_cause`, `event_type` (planned/unplanned), `priority`, `requires_road_closure`, `veh_type` (where present).
- Historical context: rolling count of incidents at the same corridor in the trailing 7/30 days, and average historical duration for that `(corridor, event_cause)` pair — this is your strongest predictive feature and also doubles as your "post-event learning" signal.
- If external data is added (see Phase 2b), join in: `is_public_holiday`, `is_major_sports_or_concert_event_nearby` (binary, from an events calendar), `rainfall_mm_last_3h` (correlates strongly with `water_logging` causes — use this as a validation chart even if not used as a model feature).

### Phase 2b — Road-network graph (used by the simulation and recommendation engine)

Build a simple graph with `networkx`:
- **Nodes** = distinct `junction` values (plus `corridor` endpoints where junction is missing).
- **Edges** = `corridor` segments connecting junctions, weighted by an assumed length and free-flow travel time (you do not have real polylines — approximate edge length from the spread of incident coordinates along that corridor, or default to a constant per corridor type: highway/ORR ≈ 2 km segments, arterial ≈ 1 km, local ≈ 0.5 km).
- This graph is reused in Phase 6 for diversion routing — build it once, save it (`networkx.write_gpickle` or similar), don't rebuild ad hoc.

## 5. Phase 3 — ML impact forecast model

Train a gradient-boosted tree model (XGBoost or LightGBM — not deep learning, this dataset is small tabular data and GBT will outperform and train in seconds):

- **Target 1 — duration_minutes**: regression, trained only on rows with a known `event_end_time`. Report MAE and R² on a held-out split, broken down by `event_cause`.
- **Target 2 — capacity_loss_fraction**: you don't have a ground-truth label for this, so derive a heuristic label from `requires_road_closure` + `priority` + `event_cause` (see the mapping table in Phase 4), train the model to predict that heuristic from the same features, and be explicit in your README that this target is calibrated, not directly observed — this is good scientific honesty for judges, not a weakness to hide.
- Produce a SHAP summary plot for the duration model — this becomes your "why did we predict this" explainability slide.
- Stretch goal (only if time remains): model `duration_minutes` as a survival problem (Cox proportional hazards via `lifelines`) so `active`-status rows contribute as censored observations instead of being dropped.

## 6. Phase 4 — Traffic flow simulation (the centerpiece — do not skip or stub this)

Implement a **Cell Transmission Model (CTM)**, the standard discretization of the Lighthill–Whitham–Richards (LWR) kinematic wave PDE `∂ρ/∂t + ∂q/∂x = 0`, with a triangular fundamental diagram.

**Why this and not something fancier:** you don't have per-vehicle trajectory data, so a microscopic simulator (SUMO, car-following models) would need a road network and calibration data you don't have. CTM only needs density and capacity per road segment, which you can derive from this dataset's `corridor`/`priority`/`requires_road_closure` fields. It is also genuinely a numerical PDE solver, so it satisfies the "differential simulation" goal honestly.

**Since the dataset gives point incidents, not road polylines**, model each affected corridor as a synthetic 1-D array of N equal-length cells (default: 25 cells × 200m = 5 km), with the incident placed at a configurable cell index so there's room for queue buildup both upstream and downstream. Document this simplification explicitly — it's a reasonable and defensible hackathon scope choice, not a hidden shortcut.

**Per-cell parameters** (set by corridor type, override per simulation run):

| Road type | Free-flow speed (km/h) | Jam density (veh/km/lane) | Capacity (veh/h/lane) |
|---|---|---|---|
| Ring road / highway (ORR, Bellary Rd) | 60 | 120 | 1800 |
| Arterial (Hosur Rd, Bannerghatta Rd, Magadi Rd) | 40 | 140 | 1500 |
| Local / non-corridor street | 25 | 160 | 900 |

**Capacity-loss mapping** (used to reduce the incident cell's capacity for the predicted duration):

| Condition | Capacity retained |
|---|---|
| `requires_road_closure = TRUE` | 5–10% |
| `priority = High` + `event_cause` in (vehicle_breakdown, accident) | 40–50% |
| `priority = Low` + `event_cause = pot_holes` | 80–90% |
| `event_cause = water_logging` | 40–60% (scale down further if rainfall data available) |
| `event_cause = tree_fall` on `Non-corridor` | treat as full closure of that local segment |

**CTM update rule** (Godunov scheme), run at a 10–15 second timestep:
- Sending function: `S_i = min(ρ_i · v_free, capacity_i)`
- Receiving function: `R_i = min(capacity_i, w · (ρ_jam − ρ_{i+1}))`, where `w` is the backward (congestion) wave speed, typically `capacity / (ρ_jam − ρ_critical)`.
- Inter-cell flow: `y_i = min(S_i, R_{i+1})`
- Density update: `ρ_i(t+Δt) = ρ_i(t) + (Δt/Δx)·(y_{i-1} − y_i)`
- Run forward from incident start until queue dissipates (capacity restored at predicted duration, then continue simulating until density returns to ~free-flow baseline).

**Outputs to capture per simulation run:**
- A space-time diagram: x-axis = cell position, y-axis = time, color = density (this is the standard traffic-engineering shockwave plot — build it with matplotlib/plotly, it is the single most convincing visual for this project).
- Max queue length (number of cells above ~80% jam density × cell length) and the timestamp it peaks.
- Total clearance time (when density returns to baseline) — this is what you validate against real data in Phase 5.

## 7. Phase 5 — Calibration & validation

For every resolved historical incident with a known duration, run the simulation with the heuristic capacity-loss parameters and compare simulated clearance time to actual `duration_minutes`. Use this comparison (grouped by `event_cause`) to tune the capacity-loss percentages in the table above — either manually or via `scipy.optimize.minimize` on mean absolute error. Produce a scatter plot of simulated vs. actual duration with an R² annotation. This plot is your proof that the simulation isn't just a pretty animation — it's grounded in the historical record.

## 8. Phase 6 — Recommendation engine

Convert simulation output into concrete, explainable recommendations:

- **Manpower** = `base_officers(corridor_type)` + `ceil(max_queue_length_km / officer_coverage_km)`, where `officer_coverage_km` defaults to 0.5 km per officer on arterials and 1 km on ring roads, plus +1 officer per affected junction node within the queue length (from the Phase 2b graph) for `priority = High` incidents.
- **Barricades** = `ceil(max_queue_length_m / barricade_spacing_m)`, `barricade_spacing_m` default 150m on arterials, 250m on ring roads, plus one barricade per side-junction within the queue extent.
- **Diversion route** = on the road graph from Phase 2b, mark the incident's corridor edge as high-cost (or removed if `requires_road_closure = TRUE`) and run shortest-path (Dijkstra via `networkx.shortest_path`) between the nearest junction upstream and downstream of the incident. Surface the top 1–2 alternate routes by travel time.

Output format per incident: a single structured object — `{predicted_duration, predicted_capacity_loss, max_queue_km, officers_recommended, barricades_recommended, diversion_route: [...]}` — this is what the dashboard renders.

## 9. Phase 7 — Interactive dashboard

Build with **Streamlit** (fastest to ship for a hackathon) or a lightweight FastAPI + simple frontend if your team prefers. Required views:

1. **Replay mode**: a timeline slider that steps through a historical day's incidents in order (simulate "real-time" using historical timestamps), plotting them on a Folium/Plotly map of Bengaluru colored by `event_cause`.
2. **Incident detail panel**: clicking an incident shows the ML prediction, the CTM space-time diagram (animated or static), and the recommendation engine's output (officers/barricades/diversion) side by side.
3. **Calibration tab**: the simulated-vs-actual duration scatter plot from Phase 5, as your validation evidence.
4. **"What-if" controls** (stretch): sliders to manually adjust `priority`/`requires_road_closure`/rainfall and re-run the simulation live, to show the system isn't a static lookup table.

## 12. Assumptions to document explicitly in the README

State clearly, in plain language: (a) the synthetic-cell corridor approximation since real road polylines aren't in the dataset, (b) the heuristic basis for `capacity_loss_fraction` since it isn't directly observed, (c) that the dataset itself is dominated by reactive micro-incidents (breakdowns, potholes) rather than true planned mega-events, and that the simulation generalizes by treating any event as "a capacity drop at a point in the network for some duration," regardless of cause. Being upfront about these is a strength in front of judges, not a weakness to hide.
